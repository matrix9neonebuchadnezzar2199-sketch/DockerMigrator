移行元 / 移行先 — まずここだけ押さえる

サイドバーは 「今どの PC でやるか」 を表すグループ名。本番では移行元 PC と移行先 PC は別マシン。lab smoke では 1 台の PC で両方の画面を順に触る（USB 移動は省略）。

本番での役割サイドバーのグループ触るメニュー・タブsmoke
移行元 — パックを作る移行元での作業「パックを書き出す」（イメージのみ）S1 の準備・S5-A
移行元 — Compose をパックにする移行元での作業「プロジェクトを選ぶ」 → 「書き出す」S2 の準備・S5-B
移行元 — 中断パックを完了移行元での作業「中断したパックを再開」 または各 Export 画面の再開S5-C
移行先 — イメージだけ取り込む移行先での作業「パックを読み込む」 → 「▶ インポート開始」S1 本体
移行先 — Compose を展開（メニューは移行元グループだが作業内容は移行先）「プロジェクトを選ぶ」 → 「取り込む」S2 本体
どちらでも共通「ロールバック」S1〜S6

「取り込む」が二つある理由

  • 移行先の 「パックを読み込む」 … イメージ中心の .dmig 専用（S1）。Compose 同梱パックはここでは読めない。
  • 「プロジェクトを選ぶ」 のタブ 「取り込む」 … Compose 同梱 .dmig 専用（S2）。本番では 移行先 PC で行うが、現行 UI ではメニューが 移行元での作業 配下。

S1 の「パックを読み込む → 取り込み」は移行先の話。 S1 手順 1 で移行元の「パックを書き出す」を触るのは、lab でパックを用意するためだけ。

UI 用語マップ（サイドバー ↔ 操作）

パックの中身移行元で書き出す移行先で読み込む備考
Docker イメージのみ移行元 → 「パックを書き出す」移行先 → 「パックを読み込む」S1
Compose プロジェクト同梱移行元 → 「プロジェクトを選ぶ」 → 「書き出す」移行先相当 → 「プロジェクトを選ぶ」 → 「取り込む」S2
取り消し——共通 → 「ロールバック」 または 「直近の操作を取り消す」

ロールバック確認ダイアログの種別 Export / Import は、書き出し／取り込みをしたパックの種別（サイドバー名とは別表記）。

実施前（必須）

NG になりやすい点: 古い Main のままだと rollback.json が生成されない・スキャンが終わらない（2a1efa4 以前）。

各シナリオは報告シートの S1〜S7（S5 は S5-A / S5-B / S5-C）と対応。順不同可。イメージのみパック（S1・S5-A）と Compose 同梱パック（S2・S5-B）の両方を通すこと。S5-C は Resume 回帰観測（スキップ可だがリリース前は 1 回推奨）。

S1 [移行先] パックを読み込む → 取り込み → ロールバック

目的移行先で イメージのみの .dmig を取り込んだあと取り消す（docker-image）
本番の流れ移行元 PC で書き出し → USB → 移行先 PC で読み込み
lab（1 台）同じ PC で「書き出し（移行元）」→「読み込み（移行先）」の順
書き出し（準備）移行元 「パックを書き出す」
読み込み（本体）移行先 「パックを読み込む」

手順

  1. 準備・移行元: 移行元での作業 → 「パックを書き出す」 → 「▶ パックを書き出す」
  2. 本体・移行先: 移行先での作業 → 「パックを読み込む」 → 「読み込み」 → 「▶ インポート開始」（インポートが完了しました。）
  3. 「ロールバック」 → 親フォルダ → 「フォルダを選んで探す」
  4. 対象パックが一覧に出る（数秒以内）。種別 Import
  5. 「ロールバック…」 → 「ロールバック実行」
  6. docker images — 取り込んだイメージが消えていること

合格

報告行: S1 [移行先] パックを読み込む → 取り込み → ロールバック: OK / NG

S2 [移行先相当] プロジェクトを選ぶ［取り込む］→ ロールバック

目的移行先で Compose 同梱 .dmig を取り込んだあと取り消す。directory は空のときのみ削除（既定）
UI の注意メニューは 移行元での作業 配下の 「プロジェクトを選ぶ」→「取り込む」（S1 の「パックを読み込む」とは別入口）

手順

  1. 準備・移行元: 「プロジェクトを選ぶ」 → 「書き出す」 → 「▶ パックを書き出す」
  2. 本体: 「プロジェクトを選ぶ」 → 「取り込む」 → 展開先 📂 → 「▶ インポート開始」
  3. 「ロールバック」 → 親フォルダをスキャン → 種別 Import → 「ロールバック実行」
  4. docker images / docker volume ls で削除確認
  5. 結果サマリで directory_not_empty 相当の文言（Compose 直後は directory 非空のためスキップが正常）

S5-A 新規 Image Export → rollback.json / ロールバック

目的新規の Image Export 直後に rollback.json が生成され、ロールバックでパックフォルダ全体が削除されること
画面移行元 「パックを書き出す」
save 場所Exporter.exportImages 末尾（fix 前から存在）

手順

  1. 移行元での作業 → 「パックを書き出す」 → イメージを 1 件以上選び 「▶ パックを書き出す」。完了したらパックパスを控える（例 F:\Docker_out\dmig-…\.dmig）。
  2. エクスプローラで <packDir>\rollback.json が存在することを確認。
  3. 「ロールバック」 → 親フォルダをスキャン → 種別 Export → 「エクスポートしたパックを削除します」→ 「ロールバック実行」。
  4. .dmig フォルダが消えていること。

python scripts/run_smoke_check.py --scan-rollback-json "F:\Docker_out"

該当パック行が OK / export / 1 のように表示されれば save 済み。

S5-C Resume Export 完了 → rollback.json（回帰観測）

目的中断したパックを再開して完了させたあとに rollback.json が生成されること（Resume 経路の回帰観測、fix=7d4f6b0）
対象Image Resume / Compose Resume の少なくとも一方（両方やれればより強い）
save 場所Exporter.resumeImagePack / ComposeExporter.resumeComposePack 末尾
前提: ベース main が 7d4f6b0 以降。python scripts/run_smoke_check.py --m10-static-check が 6 件 OK なら静的にも満たされている。

準備（中断パックの作成）

  1. 「パックを書き出す」（または 「プロジェクトを選ぶ」→「書き出す」）で書き出しを開始。
  2. 進行中にキャンセル（または npm run dev ごと停止）し、中断状態の .dmig を残す。
  3. 必要に応じて中断パックの rollback.json を削除しておく（resume 完了による生成を観測しやすくする）。

手順

  1. npm run dev を再起動。
  2. 該当ページの再開導線（「中断したパックを再開」 または Export 画面の再開）から中断パックを選び、書き出しを完了まで実行。
  3. エクスプローラで <packDir>\rollback.json を確認（再生成されていること）。
  4. 続けて 「ロールバック」 → 種別 Export → 「ロールバック実行」 でフォルダごと削除できることを確認。

観測点コマンド

python scripts/run_smoke_check.py --scan-rollback-json "F:\Docker_out"

Resume 完了直後の該当パック行が OK / export / 1 と表示されれば回帰なし。

合格

報告行: S5-C Resume Export 完了 → rollback.json: OK / NG / スキップ（中断パックなし）
